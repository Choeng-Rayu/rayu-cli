# Codebase Knowledge Graph

Rayu ships with a pre-built knowledge graph of its entire source tree. It lives at:

```
graphify-out/
  graph.json      ← the full graph (16K nodes, 59K edges)
  GRAPH_REPORT.md ← human-readable summary (communities, god nodes, surprising edges)
  graph.html      ← interactive visualisation (open in a browser)
```

The graph is generated by [`graphify`](https://github.com/21st-dev/graphify) — an AST-only extractor that
maps every function, class, import, and call-edge in the codebase.  No LLM calls are required to build it;
the semantic clustering and community detection run locally.

> **Why this exists.**  Navigating a 1 956-file TypeScript monolith is expensive in tokens.
> Every question you can answer with the graph saves a full-tree `grep`/`Read` round-trip.

## 1 ── Quick orientation

Open `graphify-out/GRAPH_REPORT.md`.  The first ~60 lines give you the most useful high-level
information without needing any tool calls:

| Section | What it tells you |
|---|---|
| **God Nodes** | The 10 most-connected functions — `logForDebugging` (1 195 edges), `logError` (591), `logEvent` (398) — these are your cross-cutting concerns |
| **Communities** | 336 semantic clusters; the named ones (Agent & Session Management, CLI Command System, Tool Execution, MCP & Plugin System, Security & Permissions, UI Components, Hooks & Events, Context & State, Type Definitions, Error Handling) map directly to the major subsystems |
| **Surprising Connections** | Infers hidden relationships like `resolveHookPermissionDecision() → canUseTool()` that aren't explicit in source but detected via semantic similarity |
| **Hyperedges** | Grouped multi-step flows (GitHub App setup, logout cache clearing, transport selection, plugin lifecycle) |
| **Knowledge Gaps** | Lists isolated nodes with ≤1 connection — potential missing edges or genuinely standalone utilities |

## 2 ── Using `graphify` as a CLI skill

When the `/graphify` skill is available, invoke it with natural-language questions.
Internally it converts your query into graph traversals and returns structured answers.

> **Tip:** Graphify queries are answered from the local `graph.json` — they are instant
> and consume no API tokens.

### 2.1 Ask a question about the codebase

```
/graphify How does the slash command /clear work — which handler, which UI component?
```

The answer will trace: command registration → handler function → state mutation → UI update,
with specific file paths and line numbers.

### 2.2 Find the path between two components

```
/graphify path src/commands/clear/conversation.ts src/state/AppStateStore.ts
```

Returns the call-chain connecting the clear-conversation command to the app state store,
including every intermediate hop.

### 2.3 Explain a symbol

```
/graphify explain getGlobalConfig
```

Returns all callers, callees, file locations, and cross-references for that function.

## 4 ── Maintenance

### Rebuild the graph after large refactors

```bash
# From the repo root
npx graphify update . --no-cluster
npx graphify cluster-only .
```

The two-step process:
1. **`update . --no-cluster`** — re-extracts AST for every file, produces a fresh `graph.json`
2. **`cluster-only .`** — re-runs community detection and writes the updated `GRAPH_REPORT.md`

### Exclude files from the graph

Add paths to `.graphifyignore` (same format as `.gitignore`) to skip generated files,
`dist/`, lockfiles, etc.

### Browse interactively

Open `graphify-out/graph.html` in any browser.  It provides:
- Search-as-you-type across all symbols
- Click any node to see its edges
- Community colour-coding to visually identify subsystems
